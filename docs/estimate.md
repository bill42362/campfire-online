# 營火直播（Campfire Online）可行性評估報告

> 撰寫日期：2026-07-19
> 評估對象：[docs/requirement.md](./requirement.md)
> 硬體前提：單機 NVIDIA 2080Ti（約 13.4 TFLOPS FP32、11GB VRAM）

---

## 0. 總結（TL;DR）

| 面向 | 結論 |
| --- | --- |
| **整體可行性** | ✅ **可行**，但屬於「橫跨多技術領域的中大型專案」，不是週末小專案 |
| **最大技術風險** | ① SWAG WebRTC 影像推流的實際協定不明；② 「擬真」燃燒 + 木堆倒塌的整合模擬 |
| **最省力路線** | 用 **成熟遊戲引擎（建議 Unity 或 UE5）** 做模擬，**OBS/ffmpeg** 負責推流，**獨立 Node 服務** 負責聊天室聚合與機器人 |
| **建議做法** | 先切一個 **MVP（風格化火焰 + 送禮加柴 + 單平台聊天室 + RTMP 推 YouTube）**，驗證互動迴圈後再逐步加擬真度 |
| **粗估工時** | MVP 約 **3–5 週**；完整需求（含擬真燃燒、木堆倒塌、雙平台、機器人）約 **3–6 個月**（單人 + Claude Code） |

核心判斷：**沒有任何一項需求是「做不到」的**，難點在於「擬真程度」與「多系統整合」的取捨。需求中每一項都可以先用「風格化 / 簡化版」達成，再視效果決定要不要投資做到「物理擬真」。這是一個非常適合漸進式開發的專案。

## 1. 系統架構總覽

建議把系統拆成三個相對獨立、可各自開發與測試的子系統：

```
┌─────────────────────────────────────────────────────────────┐
│  A. 模擬與渲染引擎 (Simulation & Render)                       │
│     火焰 / 燃燒 / 木堆物理 / 天氣光影 / 音效                    │
│     疊層 UI：送禮排行榜、整合聊天室                             │
│     ↑ 讀取狀態         ↓ 輸出畫面(window/frame)               │
└───────────────┬──────────────────────────┬──────────────────┘
                │ (共享狀態: gift、weather)  │ (畫面)
                ▼                          ▼
┌───────────────────────────┐   ┌──────────────────────────────┐
│  B. 互動與狀態服務          │   │  C. 串流輸出                   │
│  (Interaction Backend)     │   │  (Streaming)                  │
│  - 聊天室聚合(YT/SWAG)      │   │  - YouTube: RTMP (OBS/ffmpeg) │
│  - 機器人指令 /help /options│   │  - SWAG: WebRTC / pusher.js   │
│  - 送禮 → 柴火事件          │   └──────────────────────────────┘
│  - 天氣/季節/時間狀態機      │
│  - 大戶天氣複寫佇列          │
└───────────────────────────┘
```

**A 與 B 之間** 用一個簡單的即時通道（WebSocket / 本機 IPC / 共享狀態 DB）傳遞事件；**A 的畫面** 透過視窗擷取交給 C 推流。這樣切分的好處是：聊天室/機器人（B）與模擬引擎（A）可以並行開發，任何一邊改動不會互相阻塞。

## 2. 逐項可行性分析

評分說明：**可行性**（高＝技術成熟、低＝有重大未知）；**難度**（實作與調校的工程量）。

### 2.1 火焰與燃燒模擬 —— 可行性：高｜難度：高（擬真）／低（風格化）

- **風格化路線（建議先做）**：粒子系統 + shader（噪聲擾動 + 加色混合），2080Ti 綽綽有餘，可穩定 60fps。這是絕大多數「營火直播」實際採用的作法。
- **擬真路線**：GPU 流體模擬（Navier-Stokes / voxel-based combustion）。2080Ti 可跑中等解析度（例如 128³～256³）的即時流體，UE5 Niagara Fluids、Unity 或自寫 compute shader 皆可。但要「即時 + 長時間穩定運行（直播是 7×24）」需要謹慎控管記憶體與數值穩定性。
- **建議**：先粒子火焰驗證整體流程，擬真流體列為 v2 加分項。

### 2.2 木材碳化 / 發黑 / 燒紅 —— 可行性：高｜難度：中

- 作法成熟：每塊木材（或每個表面 texel）維護一個「燃燒進度 / 溫度」狀態值，用 shader 依此值在 `濕木 → 乾木 → 燒紅 → 碳黑 → 灰燼` 之間插值材質與自發光（emissive）。
- 需要一套「熱傳遞」近似（鄰近越熱、越靠近火源者燒越快）。可用簡化格點擴散，不必真解偏微分方程。
- **風險低**，主要是美術調校時間。

### 2.3 木堆脆化 / 餘燼掉落 / 倒塌 —— 可行性：中｜難度：高

- 這是**除了 SWAG 推流以外的第二大難點**。需要剛體物理引擎（PhysX / Jolt / Bullet，Unity/UE5 內建）。
- 「燒到一定程度 → 結構強度下降 → 失去支撐 → 倒塌」需要把「燃燒狀態」與「物理約束」耦合：例如燒穿的木材移除碰撞約束、觸發鄰塊重新落下。
- 難在**即時 + 穩定**：物理引擎長時間運行可能累積數值抖動；倒塌瞬間大量碰撞可能掉幀。
- **建議**：MVP 先用「預設的幾段式塌陷動畫 / 觸發式」近似，真物理塌陷列為 v2。

### 2.4 天氣對燃燒的影響（雨濕底部、雪濕頂端燒得慢）—— 可行性：高｜難度：中

- 本質上是 2.2 的延伸：在木材的「濕度」狀態上加一個由天氣驅動的空間分布（底部/頂端），濕度越高燃燒速率越低、發煙越多。
- 純狀態邏輯 + shader 表現，**技術風險低**。

### 2.5 劈啪聲音效 —— 可行性：高｜難度：低～中

- 最簡單：預錄劈啪音效庫，依「燃燒強度 / 事件（餘燼爆裂）」隨機觸發播放並做空間化。**低難度、效果已足夠好**。
- 進階：程序化音效合成（依火勢即時生成），屬加分項。
- **注意**：直播音訊需與畫面一起進 OBS，這步不難但別漏。

### 2.6 白天/黑夜 + 四季 + 五種天氣（20 天循環）—— 可行性：高｜難度：低

- 純粹是一個由時鐘驅動的**狀態機**：
  - 白天黑夜：依 CST 現在時間映射到天光顏色/亮度/太陽角度。
  - 季節：`floor(dayIndex) % 4`；天氣：`floor(dayIndex) % 5`；20 天大循環＝4 與 5 的最小公倍數。
- 表現面（天光顏色、環境光、天氣粒子如雨雪）是美術與 shader 工作，邏輯本身**非常簡單、風險極低**。
- **需釐清**：需求說「四季分四天輪流、五種天氣分五天輪流」——這是**加速模擬**（一天換一季/一種天氣），要確認這是刻意設計（直播節奏需要），我判斷是。

### 2.7 送禮 → 加柴 / 燒完升白煙 / 排行榜 —— 可行性：高｜難度：低～中

- 送禮事件來自各平台 webhook / API（見 2.9）。收到後依價值換算成「投入木材量」，在模擬中生成新木材物件。
- 「燒完 → 白煙」：當可燃物總量趨近 0，切換到白煙粒子狀態。純狀態判斷。
- 排行榜：對送禮事件累加、排序，畫面疊層渲染。**風險低**，主要是要有持久化（重啟不歸零）。

### 2.8 聊天室整合（多平台 → 單一聊天室疊層）—— 可行性：高｜難度：中

- **YouTube**：`YouTube Live Streaming API` 的 `liveChatMessages`。注意 **API quota 限制與輪詢延遲**（免費額度每日 10,000 units，長時間直播需申請提額或控管輪詢頻率），這是實務上要規劃的點。
- **SWAG**：透過 **pusher.js** 訂閱聊天頻道即可拿到訊息（pusher 是即時訊息服務，適合聊天，**不是**影像通道）。需要 SWAG 提供頻道名稱/認證方式。
- 兩邊訊息正規化成統一格式（來源平台、暱稱、內容、時間）後合流，渲染到畫面疊層。**技術成熟，風險中等**，主要不確定性在各平台的 API 存取權限與文件。

### 2.9 機器人指令（/help、/options、大戶複寫天氣）—— 可行性：高｜難度：中

- 指令解析很簡單（前綴比對 + 參數）。難點在**回覆管道與權限**：
  - 回覆訊息需各平台的「發送訊息」API（YouTube 送訊息也吃 quota）。
  - 「大戶」判定需綁定送禮累計額 → 需要身分辨識（platform user id）與門檻設定。
  - 「複寫未來 24hr 天氣」＝往 2.6 的狀態機塞一個帶時效的 override，覆蓋預設排程。邏輯清楚、風險低。
- **風險中等**，集中在平台 API 權限與濫用防護（rate limit、誰能下指令）。

### 2.10 串流輸出 —— YouTube 可行性：高｜SWAG 可行性：中（有未知）

- **YouTube (RTMP)**：**最成熟的一環**。遊戲引擎畫面 → OBS（或 ffmpeg）→ RTMP push 到 YouTube。零技術風險。
- **SWAG (WebRTC / pusher.js)**：**本專案最大的未知**。
  - WebRTC 影像**推流（ingest）** 不像 RTMP 有統一標準，需要 SWAG 的 signaling 伺服器 / SFU 端點與認證流程細節。
  - `pusher.js` 一般用於**聊天等即時訊息**，不是影像；需確認 SWAG 的影像上行到底走哪個路徑（可能是自訂 WebRTC ingest，或某個 SDK）。
  - **行動建議**：**盡早向 SWAG 取得推流技術文件 / SDK / 測試帳號**，這一項的答案會顯著影響架構（例如是否需要一個把 OBS 輸出轉成 WebRTC 上行的橋接服務，如 `mediasoup` / `ffmpeg + whip`）。

## 3. 技術選型建議

| 子系統 | 建議方案 | 備選 | 理由 |
| --- | --- | --- | --- |
| 模擬 / 渲染 | **Unity**（VFX Graph + 內建物理）或 **UE5**（Niagara Fluids + Chaos 物理） | 自寫 WebGPU/three.js + compute shader | 引擎內建粒子/流體/剛體物理，省下巨量底層工作；2080Ti 支援良好 |
| 互動後端 | **Node.js / TypeScript** 服務 | Python | 生態成熟、與 pusher.js / YouTube API 相容好，即時 WebSocket 方便 |
| 聊天室聚合 | YouTube Live Chat API + pusher.js client | 第三方聚合服務 | 直接對接、可控 |
| YouTube 推流 | **OBS Studio**（可用 obs-websocket 自動化）或 **ffmpeg** | 引擎內建 RTMP | 最成熟穩定 |
| SWAG 推流 | **待 SWAG 文件確認**（可能 WHIP/WebRTC 橋接） | mediasoup / Janus | 現階段未知，需先釐清 |
| 狀態 / 持久化 | SQLite 或 Redis | Postgres | 排行榜、送禮累計、天氣 override 需持久化 |

> 若團隊更熟 Web 技術、且能接受「風格化而非物理擬真」，用 **three.js / WebGPU 在瀏覽器內** 做也完全可行，且推流與 WebRTC 反而更自然（瀏覽器原生支援 WebRTC）。這在「SWAG 走 WebRTC」的前提下可能是加分項，值得列為架構分歧點評估。

## 4. 主要風險與對策

| # | 風險 | 等級 | 對策 |
| --- | --- | --- | --- |
| R1 | **SWAG WebRTC 推流協定不明** | 🔴 高 | 專案啟動第一件事就取得 SWAG 推流文件/SDK/測試環境；在架構定案前先做一個 30 分鐘的推流連通 PoC |
| R2 | **擬真燃燒 + 木堆倒塌整合的工程量被低估** | 🟠 中高 | 明確分 v1（風格化/觸發式）與 v2（物理擬真）；先交付 v1 |
| R3 | **7×24 長時間穩定性**（記憶體洩漏、物理數值累積、API token 過期） | 🟠 中 | 從一開始就設計「可自動重啟 / 狀態可回復」；壓力測試連續運行 24h+ |
| R4 | **YouTube Chat API quota 與延遲** | 🟡 中 | 控管輪詢頻率、必要時申請提額、機器人回覆做節流 |
| R5 | **平台 API 權限 / 送禮 webhook 取得** | 🟡 中 | 及早確認 YouTube、SWAG 的送禮事件如何取得（API? webhook? 畫面解析?） |
| R6 | **「盡可能全用 Claude Code 開發」對整合類、需外部帳號測試的工作幫助有限** | 🟡 中 | Claude Code 對模擬/邏輯/前端幫助極大；但推流連通、平台授權等需真人拿帳號測試，預留這部分人力 |

## 5. 建議里程碑與粗估工時

> 前提：**單人開發者 + Claude Code 高度輔助**。工時為概略區間，實際依「擬真度目標」浮動極大。

### M0 — 技術探底（3–5 天）
- SWAG 推流 PoC（R1）、YouTube RTMP 推流 PoC、YouTube/SWAG 聊天與送禮事件取得可行性確認。
- **產出**：一份「平台對接可行性確認」，據此鎖定架構。

### M1 — 可玩 MVP（約 2–3 週）
- 風格化火焰 + 木材（粒子 + 基本燃燒狀態）。
- 送禮 → 加柴、燒完 → 白煙。
- 單平台（先 YouTube）聊天室疊層 + 送禮排行榜。
- OBS → RTMP 推 YouTube 全鏈路打通。
- **產出**：一個真的能開直播、觀眾送禮會加柴的最小版本。

### M2 — 環境與雙平台（約 2–4 週）
- 白天黑夜 + 四季 + 五天氣（20 天循環）狀態機與天光/雨雪表現。
- SWAG 推流 + SWAG 聊天併入統一聊天室。
- 機器人指令 `/help` `/options`，大戶天氣複寫。
- 持久化（排行榜、送禮累計、override）。

### M3 — 擬真化（約 4–8 週，可切分逐步交付）
- 木材碳化/發黑/燒紅材質、濕度對燃燒的影響。
- 劈啪音效系統。
- 木堆脆化 / 餘燼掉落 / 倒塌（物理）。
- 進階：GPU 流體火焰、程序化音效、更多可燃物（炭、樹枝、樹葉）。

**合計**：可用 MVP 約 **3–5 週**；接近完整需求約 **3–6 個月**。

## 6. 需要與需求方釐清的問題

1. **擬真 vs 風格化**：火焰與燃燒要「物理擬真」還是「好看的風格化」即可？這決定工時級距。
2. **SWAG 推流**：影像上行的實際協定 / SDK / 端點為何？有無測試帳號與文件？（最關鍵）
3. **送禮事件來源**：YouTube 與 SWAG 的送禮事件如何取得——官方 API、webhook、還是需畫面解析？
4. **機器人回覆權限**：機器人是否能以官方身分在各平台聊天室發言？權限如何取得？
5. **「大戶」定義**：以什麼門檻、跨不跨平台累計、override 的優先權與衝突規則？
6. **運行模式**：是 7×24 常駐直播，還是定時開播？這影響穩定性投資。
7. **執行環境**：2080Ti 是開發機還是正式播出機？播出時模擬、推流、聊天服務是否同一台（資源競爭）？

---

## 7. 結論

**這個專案技術上完全可行，且非常適合以「漸進式擬真」的方式開發。** 需求裡沒有任何「不可能」的項目，真正決定成敗與工時的是兩件事：

1. **SWAG WebRTC 推流的實際協定** —— 唯一的高等級未知，應在專案第一週就消除。
2. **擬真程度的取捨** —— 每一項模擬都可以先做風格化版本上線，再決定是否值得投資做到物理擬真。

建議路線：**先用成熟遊戲引擎快速做出「送禮加柴 + 單平台聊天室 + RTMP 推流」的可玩 MVP**，讓互動迴圈先跑起來、驗證觀眾反應，再把火焰擬真、木堆倒塌、雙平台、機器人等逐步疊上去。這樣風險最低、回饋最快，也最能發揮 Claude Code 在模擬邏輯與前端上的輔助效率。
